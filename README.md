# Coturn on StartOS

> **Upstream repo:** <https://github.com/coturn/coturn>

[Coturn](https://github.com/coturn/coturn) is a mature, standards-compliant TURN and STUN server. It relays audio and video for real-time applications (video calls, messaging) when peers are behind NAT or restrictive firewalls. This package runs a single, general-purpose Coturn instance that other StartOS services — such as Jitsi Meet — depend on for connectivity.

| Variable            | Purpose                                                                                             |
| ------------------- | --------------------------------------------------------------------------------------------------- |

## Volume and Data Layout

| Volume | Mount Point       | Contents                                                                        |
| ------ | ----------------- | ------------------------------------------------------------------------------- |
| `main` | `/var/lib/coturn` | `turnserver.conf`, `cert.pem`, `pkey.pem`, `store.json` (shared secret), `turndb` |

`turnserver.conf`, `cert.pem` and `pkey.pem` are regenerated on every start from the enabled public domain, its StartOS-managed certificate, the public IP, and the shared secret. `store.json` holds the generated TURN secret and persists across restarts.

---

## Installation and First-Run Flow

1. On install, the package generates a random shared TURN secret and stores it in `store.json`.
2. Coturn cannot serve traffic until the user **adds and enables a public (clearnet) domain** for the TURN interface. Until then coturn idles and the **Public Domain** health check fails.
3. Once a public domain is enabled, StartOS provisions a TLS certificate for it, the config is written, and coturn starts.

---

## Configuration Management

There is no user-facing configuration. `turnserver.conf` is generated by the package:

- `realm` / `server-name` — the enabled public domain
- `external-ip` — the host's public IPv4 address(es)
- `cert` / `pkey` — the StartOS-managed certificate for the domain (Coturn terminates TLS/DTLS itself)
- `listening-port` `3478`, `tls-listening-port` `5349`
- `min-port` / `max-port` — the relay UDP range (`49152`–`49651`)
- `use-auth-secret` + `static-auth-secret` — TURN REST API authentication

## Network Access and Interfaces

| Interface        | Port(s)        | Protocol        | Purpose                                          |
| ---------------- | -------------- | --------------- | ------------------------------------------------ |
| TURN / STUN      | 3478           | UDP + TCP       | STUN binding and TURN relay                       |
| TURN over TLS    | 5349           | TCP (TLS) + UDP (DTLS) | Encrypted TURN (`turns:`)                  |
| TURN Relay Ports | 49152–49651    | UDP             | Media relay allocations (bound via `bindPortRange`) |

TURN requires **inbound reachability from the public internet** — peers connect directly to the server. The user must add and enable a **public clearnet domain** and forward ports `3478`, `5349` and the relay UDP range (`49152`–`49651`) at their router/ISP. Coturn is not usable over Tor or LAN only.

---

## Authentication and the Consumer Contract

Coturn uses the **TURN REST API** (shared-secret) model: one `static-auth-secret` from which each consuming service mints short-lived, per-session credentials (`username = <expiry-timestamp>`, `password = base64(HMAC-SHA1(secret, username))`).

A dependent StartOS package integrates with coturn by:

1. **Hostname / port** — reading coturn's public domain from the `turn` interface via `sdk.host.get(effects, { hostId: 'turn', packageId: 'coturn' })`.
2. **Shared secret** — mounting coturn's `main` volume read-only and reading `TURN_SECRET` from `store.json`.

## Backups and Restore

**Included in backup:**

- `main` volume (includes `store.json`, so the shared secret — and therefore dependent services — keep working after a restore)

**Restore behavior:** Volume is fully restored before the service starts.

## Health Checks

| Check         | Method                                | Behavior                                                                                     |
| ------------- | ------------------------------------- | -------------------------------------------------------------------------------------------- |
| TURN Server   | Port listening (3478)                 | Success once coturn is listening; `disabled` while it waits for a public domain              |
| Public Domain | Enabled public-domain check           | **Fails** until a public clearnet domain is added and enabled; `loading` while the cert provisions |

## Limitations and Differences

1. **A public clearnet domain is required.** TURN relies on inbound reachability from arbitrary peers, so coturn cannot run Tor-only or LAN-only.
2. **Relay capacity is bounded by the port range.** The relay range is 500 UDP ports (`49152`–`49651`), roughly 500 concurrent relay allocations — ample for a self-hosted deployment but not unlimited.
3. **Shared-secret (TURN REST API) authentication only.** Static per-user long-term credentials are not configured.

---

## What Is Unchanged from Upstream

The Coturn binary and its behavior are exactly as shipped in the upstream `coturn/coturn` image. StartOS only generates the configuration, certificate and key files it runs with.

## Quick Reference for AI Consumers

```yaml
package_id: coturn
image: coturn/coturn
architectures: [x86_64, aarch64]
runs_as: nobody
volumes:
  main: /var/lib/coturn
interfaces:
  turn: { ports: [3478], protocol: udp+tcp }
  turns: { ports: [5349], protocol: tls+dtls }
  turn-relay: { ports: 49152-49651, protocol: udp }
auth: turn-rest-api (use-auth-secret / static-auth-secret)
consumer_contract:
  hostname: sdk.host.get(effects, { hostId: 'turn', packageId: 'coturn' })
  shared_secret: mount coturn main volume read-only, read store.json TURN_SECRET
startos_managed_env_vars:
  coturn: [COTURN_CONFIG_REV]
actions: none
dependencies: none
requires_public_domain: true
```
